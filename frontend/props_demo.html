§ Props paper — section 2.3 constraining adversarial inputs
1
FORGED DOCUMENT
Attempt: forged medical license PDF submitted directly
This credential was submitted directly to the attestation endpoint — not fetched from nysed.gov through an authenticated TLS session inside the enclave. Props requires all data to enter through the oracle layer.
2
FAKE REGISTRY
Attempt: fake medical board website with real TLS certificate
The oracle is pinned to the certificate fingerprint of the real NY State Medical Board (nysed.gov). A valid TLS certificate from a different authority is rejected immediately. Fingerprints did not match.
3
DATA INTERCEPTION
Attempt: real credential intercepted and modified in transit
A real medical license (general practitioner) was intercepted and the specialty field swapped to "cardiologist." The enclave recomputed the data hash and compared it against the oracle-attested hash. They did not match.
Why this is architectural, not a filter
A valid Props certificate is a cryptographic proof that data arrived from the real authoritative source, unmodified, processed inside hardware that nobody can tamper with. That proof cannot be faked without breaking the enclave or compromising the authoritative source itself.

This is the paper's section 2.3 adversarial input constraint implemented as the core security property of the system.
