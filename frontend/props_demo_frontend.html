§ Props paper — section 2.3 constraining adversarial inputs
1
FORGED DOCUMENT
Attempt: forged medical license PDF submitted directly
REJECTED — data not oracle-authenticated.
This credential was submitted directly to the attestation endpoint. It was not fetched from nysed.gov through an authenticated TLS session inside the enclave. The pipeline requires all data to enter through the oracle layer. No certificate issued.
2
FAKE REGISTRY
Attempt: fake medical board website with real TLS certificate
REJECTED — endpoint fingerprint mismatch.
The oracle is pinned to the certificate fingerprint of the real NY State Medical Board (nysed.gov). The attacker's server presented a valid TLS certificate but from a different authority. Fingerprints did not match. Oracle refused the connection. No certificate issued.
3
DATA INTERCEPTION
Attempt: real credential intercepted and modified in transit
REJECTED — enclave tamper detection triggered.
The attacker had a real medical license (general practitioner) and intercepted the data stream between the oracle and the enclave to change the specialty field to "cardiologist." The enclave computed a hash of the incoming data and compared it against the oracle-attested hash. They did not match. Certificate refused.
Why this is architectural, not a filter
A valid Props certificate is a cryptographic proof that data arrived from the real authoritative source, unmodified, processed inside hardware that nobody can tamper with. That proof cannot be faked without breaking the enclave or compromising the authoritative source itself. There is no configuration to disable, no exception to request, no edge case to exploit.

This is the paper's section 2.3 adversarial input constraint implemented as the core security property of the system.
